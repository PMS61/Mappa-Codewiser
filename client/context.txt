The unified idea
A task scheduler that doesn't just ask "when are you free?" but "when is your brain ready for this specific task?"
You input your tasks. The system estimates the cognitive load of each one, maps it against your mental bandwidth curve across the day, and produces a schedule where hard tasks land at your peak hours and light tasks fill your recovery windows — automatically, deterministically, with full reasoning shown.

## Features mapped to intelligence approaches

---

### Rules Engine

**Fixed task blocking**
If a time slot contains a fixed task → block it entirely from scheduler. No override possible.

**Recovery block insertion**
If cumulative CL in any 3-hour window exceeds threshold → insert break automatically. Rule fires deterministically.

**Deadline enforcement**
If task has deadline and no valid slot exists before it → surface conflict. Never silently drop.

**Sacrifice gate**
If user requests sacrifice → only permitted if task CL > 5.0 and deadline is > 6 hours away. Below those values, sacrifice is blocked — the task is already light enough.

**Deadline extension gate**
If deadline extension requested → only permitted if task priority is Normal or Low. High priority tasks cannot be extended — forces the user to confront the consequence instead.

**Task switching limiter**
If 3 or more different subject tags appear in consecutive 90-minute window → flag as excessive context switching. Reschedule to group subjects.

**Same-task duration cap**
If a single task occupies more than 90 consecutive minutes → force-split into two sessions with a minimum 15-minute break between them.

**Recreational task energy gate**
If energy level drops below -1 threshold mid-schedule → recreational task automatically promoted from optional to scheduled. Rule fires without user input.

**Productive hours protection**
If a high CL task (CL > 7.0) is being placed outside a tracked productive hour → block placement and route to nearest available productive window.

---

### Scoring System

**Cognitive load score per task**
```
CL = base_difficulty × duration_weight × deadline_urgency
     × task_type_multiplier × priority_weight
```
Every variable is explicit. Score shown on every task card with full breakdown.

**Energy delta score**
Recreational tasks carry a negative CL value — they restore bandwidth rather than consume it. A 30-minute walk = -2.0 CL. A 20-minute game = -1.5. User sets the energy recovery value when adding the task. Net CL of a window = sum of all task CLs including recreational.

**Slot fitness score**
When placing a task, every available slot gets a fitness score:
```
fitness = bandwidth_at_slot - task_CL
          + productive_hour_bonus
          - context_switch_penalty
          - proximity_to_deadline_bonus
```
Highest fitness slot wins. Reasoning chain shows the top 3 candidates and why the winner was chosen.

**Sacrifice score**
When user requests sacrifice, system calculates exactly how much quality is being traded:
```
sacrifice_impact = (original_CL - reduced_CL) / original_CL × 100
```
Shown as "You are accepting a 32% reduction in depth for this task."

**Weekly productivity score**
End-of-week score across five dimensions — schedule adherence, CL balance, context switching rate, deadline hit rate, energy consistency. Each dimension scored 0–100, weighted into a composite. Fully traceable per dimension.

### State Machine

**Task lifecycle states**
```
Unscheduled → Scheduled → In Progress → Completed
                                      → Skipped → Rescheduled
                                      → Sacrificed
                                      → Deadline Extended
```
Every transition is guarded. A task cannot move to Completed without passing through In Progress. A Sacrificed task cannot be un-sacrificed once marked — it stays in history.

**Energy level state**
```
Exhausted (-2) → Depleted (-1) → Baseline (0) → Energised (+1) → Peak (+2)
```
Transitions triggered by user input at day start or mid-day update. Each state applies a different bandwidth multiplier. Recreational task completion can trigger an upward transition — completing a -2.0 CL recreational block can shift state from Depleted to Baseline automatically.

**Day phase state**
```
Planning → Active → Wind-down → Complete
```
Planning: scheduler runs, user reviews. Active: tasks being executed, adaptive re-runs enabled. Wind-down: only low CL tasks permitted in remaining slots. Complete: productivity report generated.

**Burnout risk state**
```
Safe → Watch → Warning → Critical
```
Driven by 7-day rolling CL total. Each state triggers different scheduler behaviour — Safe runs normally, Watch surfaces a suggestion, Warning forces at least one rest day in the next 3 days, Critical blocks high CL tasks from being scheduled until rolling total drops.

**Productive hours learning state per slot**
```
Unknown → Observed → Tracked → Confirmed
```
A time slot starts Unknown. After the user completes a task in that slot 3 times → Observed. After 7 times → Tracked. After 14 consistent completions → Confirmed productive hour. Confirmed slots get the productive hour bonus in fitness scoring.

---

### Data Models

**96 × 7 productive hour matrix**
The week is divided into 15-minute units. 96 slots per day × 7 days = 672 cells. Each cell stores:
- `scheduled_CL` — what the scheduler placed there
- `actual_completion` — boolean, did the user complete the task
- `energy_state` — user's energy level at that time
- `productive_hour_score` — rolling average of completion rate for this slot across all weeks

This matrix is the core data structure the entire system reads from and writes to. It gets more accurate every week. The scheduler reads confirmed productive hours from this matrix when computing slot fitness scores.

**Bandwidth curve model**
Default ultradian curve stored as a 96-point array — one bandwidth value per 15-minute slot. User's energy level state applies a scalar multiplier to the entire array at schedule time. Personal peak windows, once confirmed by the matrix, override the default curve values for those slots.

**Task type multiplier table**
Fixed lookup table mapping task types to CL multipliers:

| Task type | Multiplier |
|---|---|
| Learning new concept | 1.4 |
| Problem solving | 1.3 |
| Writing / drafting | 1.1 |
| Revision | 0.9 |
| Reading | 0.8 |
| Administrative | 0.6 |
| Recreational | -0.5 to -2.0 |

Deterministic by design. No inference.

---

### Decision Tree

**Conflict resolution router**
When a task cannot be placed, a decision tree routes to the right resolution:

```
Can it fit tomorrow without violating deadline?
  Yes → defer to tomorrow, notify user
  No → Is deadline negotiable (Normal/Low priority)?
    Yes → offer deadline extension
    No → Is CL > 5.0?
      Yes → offer sacrifice threshold
      No → escalate to user — manual resolution required
```

**New task intake classifier**
When a task is added, a decision tree classifies it:
```
Does it have a deadline?
  Yes → Is deadline today?
    Yes → urgent pool, re-run scheduler immediately
    No → deadline pool, next scheduler run
  No → Is it recreational?
    Yes → energy pool, placed in troughs
    No → floating pool, placed in leftover capacity
```

**End-of-day review router**
```
Adherence > 80%? → Positive report, minor suggestions
Adherence 50-80%? → Mixed report, identify pattern in skipped tasks
Adherence < 50%? → Flag overloading, suggest reducing tomorrow's CL budget
Burnout state Warning or Critical? → Override tomorrow's schedule with forced light day
```

---

### Retrieval and Ranking

**Best slot finder**
For each unplaced task, retrieve all available slots, rank by fitness score, place in the top-ranked slot. The ranking is multi-dimensional — bandwidth fit, productive hour bonus, context switch penalty, deadline proximity bonus. Top 3 candidates shown in reasoning chain so user understands why the winner was selected.

**Recreational task selector**
When energy drops below threshold and a recovery block needs filling, retrieve all recreational tasks from the user's library ranked by energy recovery value per minute. Highest ranked task that fits the available slot duration is placed automatically.

**Weekly plan optimiser**
At the start of each week, retrieve all tasks with deadlines in the next 7 days, rank by urgency × CL × priority, distribute across days to balance daily CL totals. Days with fixed task overload get fewer scheduled tasks automatically.

---

### Vector Search and Embeddings (supporting infrastructure)

**Task similarity detection**
Embed task titles and descriptions offline. At task creation, find the most similar past tasks from the user's history. Use their actual completion time and CL experience from similar past tasks to improve the duration and difficulty estimates for the new one. "You estimated 45 minutes for a similar Graph Theory problem in Week 3 but it took 90. Adjusting estimate."

**Subject clustering**
Embed subject tags to understand semantic proximity. Tasks from semantically similar subjects (Linear Algebra and Machine Learning) get a smaller context switch penalty than truly unrelated subjects (Chemistry and Software Engineering). Computed offline, stored as a similarity matrix, read deterministically at schedule time.

## Learning domain features specifically

**Concept difficulty progression tracker**
Within a subject, tasks are tagged with concepts. State machine tracks mastery per concept — Unseen → Attempted → Practicing → Confident. CL multiplier for a concept decreases as mastery state advances. A Confident concept gets 0.8× the CL of an Unseen one. Encourages revisiting rather than avoiding hard topics.

**Study session sequencing rule**
Rules engine enforces spaced repetition logic — if a concept was last studied more than 5 days ago and is not yet Confident, it gets an urgency bump equivalent to a 2-day deadline. Surfaces forgotten material before it becomes a crisis.

**Pre-exam load balancing**
If an assignment or exam deadline is within 3 days, the scheduler enters exam mode for that subject — reserves the first productive window of each remaining day for that subject exclusively, reduces recreational scheduling to one block per day, suppresses new floating task additions for that subject.

**Cognitive load learning curve**
The system tracks whether high CL tasks placed in productive windows actually get completed vs skipped. Over time it adjusts the default CL multipliers for each task type based on the user's personal completion patterns. Not ML — just a running weighted average updated on each completion event.

---

## Final productivity report

Generated at end of each day and week. Contains:

- **Schedule adherence rate** — tasks completed on time / tasks scheduled
- **CL balance score** — how evenly CL was distributed across windows
- **Productive hours accuracy** — did confirmed productive hours actually produce completions
- **Context switching score** — how many subject transitions occurred vs optimal
- **Energy management score** — did recreational tasks prevent threshold breaches
- **Deadline hit rate** — deadlines met / deadlines due
- **Burnout risk trend** — 7-day rolling CL chart with threshold line
- **Top insight** — single most actionable finding, rules-generated: "You consistently skip tasks scheduled after 8PM. Recommend setting Wind-down state at 8PM."

Every number in the report traces to a specific data point in the 96×7 matrix. Nothing is inferred.


UI Design Guidelines

Build Axiom,  a cognitive-aware task scheduler web app
The aesthetic is "Systems Typography" — the visual language of a peer-reviewed
engineering paper meets a Swiss grid. The UI must feel like the scheduling engine
drew its own interface using only a typewriter and a ruler. No decoration, pure
information architecture.

COLOR PALETTE (Strictly enforced)
- Background: Cream-white #FAF7F2
- Primary Text / Rules: Ink-black #1A1A1A
- Grid Lines / Separators: 0.5px #D4C9B8 (rulings only, never for boxes)
- Accent (Absolute Minimum Use): Deep Vermillion #C0392B. Used ONLY for:
  1. The cognitive bandwidth sparkline.
  2. CL threshold breach indicators.
  3. Scheduling conflicts/overflow states.

TYPOGRAPHY (The core aesthetic)
- Playfair Display: Used EXCLUSIVELY for the main Date display (e.g., "Thursday,
  April 3") and top-level section headers (e.g., "REASONING CHAIN").
- IBM Plex Mono: Used for ABSOLUTELY EVERYTHING ELSE. Task names, time slots,
  CL values, UI labels, buttons. Task names are not marketing copy; they are data
  points rendered in monospace.

LAYOUT & GRID
- Strict, unyielding vertical and horizontal alignment.
- No shadows, no gradients, no rounded corners, no card elevation.
- Separation is achieved purely through 0.5px ruled lines and generous, intentional
  white space.

TASK RENDERING (Data-as-Structure)
- Tasks render as column entries in the daily schedule.
- HEIGHT PROPORTIONAL: The vertical height of a task block MUST be strictly
  proportional to its duration (e.g., a 60-min task is exactly twice as tall as
  a 30-min task).
- CL HEADLINE: The Cognitive Load (CL) score is typeset as a large, unapologetic
  number in the top-left corner of the task block (e.g., "8").
- CL WEIGHT: The top border of the task block scales in thickness from 1px
  (CL 1) to 4px (CL 10) in ink-black, acting as a visual weight indicator.
- Task metadata (time, dependencies) is set in small, tightly tracked IBM Plex Mono.

THE BANDWIDTH CURVE
- A thin, unsmoothed vermillion polyline (#C0392B) spanning the top of the daily
  columns. It should look like a raw data readout, not an Illustrator curve.
- It strictly maps the aggregate CL demand against the user's defined bandwidth
  limit.

THE REASONING CHAIN PANEL (The Hackathon Core)
- This is not a sidebar; it is a footer or adjacent column that reads like a
  mathematical proof or a legal ruling.
- Format: Numbered axioms leading to an inescapable conclusion. No prose, no
  filler words.
- Clean indentation using the 0.5px ruled lines.
- Example output format:
  1. CL-demand("Write Report") = 8
  2. Current Slot (09:00) bandwidth_remaining = 3.5
  3. RULE: IF demand > remaining, defer to next peak window
  4. Next peak window = 14:00 (bandwidth = 9.0)
  5. CONFLICT: "Team Sync" (CL=4) already occupies 14:00
  6. APPLY RULE: Higher CL priority resolves conflict
  7. → ACTION: Schedule "Write Report" at 14:00. Bump "Team Sync" to 15:30.

INTERACTIONS
- Hovering over a task highlights its corresponding logic steps in the Reasoning
  Chain panel using a subtle #FAF7F2 background invert (or light gray).
- Adding/editing a task instantly re-renders the polyline and appends a new
  numbered logic block to the chain. The user sees the system "think."

ABSOLUTE PROHIBITIONS
- No emoji. No icons (use text like [!] for warnings).
- No smooth CSS transitions or animations. State changes must be instant and crisp.
- No marketing language in the UI (e.g., no "Boost your productivity!", only
  "Cognitive Load Threshold: 7").
